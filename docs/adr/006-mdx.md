# ADR-006 · MDX para projetos

**Data:** 2026-07-12  
**Status:** Aceito

---

## Contexto

Os projetos do portfólio podem conter conteúdo rico: diagramas de
arquitetura, snippets de código, imagens de documentação. Markdown
puro suporta imagens e código, mas não suporta componentes interativos
ou layouts customizados dentro do corpo do texto.

O manifesto e o about, por outro lado, são exclusivamente textuais.
Não há necessidade de componentes embedados nesses documentos.

Precisamos decidir:
- Usar MDX para todo o conteúdo ou apenas para projetos?
- Quais componentes disponibilizar no escopo MDX?
- MDX compila em build time (via Astro) ou runtime?

---

## Decisão

**MDX habilitado apenas para a coleção `projects/`.** Manifesto e
About usam markdown puro (`.md`).

Componentes disponíveis no escopo MDX de projetos:
- `Image` — tratamento duotone automático.
- `Quotation` — bloco de citação estoica.
- `Text` — forçar voz tipográfica onde necessário.

Nenhum componente interativo no MDX. Sem islands. Sem scripts.

Compilação via `@astrojs/mdx` em build time — zero runtime de MDX.

---

## Alternativas rejeitadas

| Alternativa | Razão da rejeição |
|---|---|
| **Markdown puro para projetos** | Limita conteúdo rico. Diagrama de arquitetura em markdown puro é apenas `![alt](img.png)` — não permite tratamento duotone automático, legenda no formato `FIG. XX`, crop marks. O componente `Image` do MDX encapsula todo o tratamento. |
| **MDX para todo o conteúdo** | Complexidade desnecessária no manifesto e about, que são texto puro. MDX exige parsing adicional e expande a superfície de bugs. |
| **MDX com componentes interativos** | Violaria o princípio de islands — MDX não é o lugar para hidratação. Componentes interativos pertencem às seções Astro, não ao conteúdo markdown. |
| **Remark/Rehype plugins em vez de MDX** | Criar plugins de remark para tratamento de imagem seria frágil e difícil de manter. MDX com componentes Astro é mais direto e type-safe. |

---

## Consequências

**Positivas:**
- Projetos podem usar `<Image treatment="blueprint" ratio="4:3" caption="FIG. 01 — ..." />` diretamente no markdown.
- Snippets de código em markdown com syntax highlighting (via plugin de remark ou CSS).
- Separação clara: conteúdo textual → `.md`, conteúdo rico → `.mdx`.

**Negativas:**
- Dependência adicional: `@astrojs/mdx`.
  Mitigação: é uma integração oficial do Astro, mantida pelo core team.
- Dois formatos de conteúdo (`.md` e `.mdx`) exigem que o autor saiba
  qual usar.
  Mitigação: documentado. Projetos = `.mdx`, Manifesto/About = `.md`.

**Ações:**
- Adicionar `@astrojs/mdx` ao `astro.config.ts`.
- Definir 3 componentes exportados para escopo MDX (`Image`, `Quotation`, `Text`).
- Migrar projetos existentes de `.md` para `.mdx` se necessário.
